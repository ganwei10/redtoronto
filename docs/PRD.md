# PRD · 产品需求文档（Product Requirements Document）

> 产品：**红多营 RedToronto** MVP
> 版本：v0.1　日期：2026-07-18　关联：MRD、项目架构文档

---

## 1. 产品概述

把「**博主匹配 + Campaign 管理 + 效果看板**」做成可运行的 AI 驱动 Web 应用，支撑前 3–6 个月业务验证。

- **一句话**：商户填需求 → AI 出 Brief → 系统匹配博主 → AI 出内容 → 投放（代理商侧）→ 看板归因 → AI 复盘。
- **技术基线**：Next.js 15（App Router）+ TypeScript + Tailwind；Postgres + pgvector；Claude 做 AI；Vercel 部署。

---

## 2. 目标与范围

**目标**：验证博主匹配与效果归因的核心闭环，降低人工匹配成本，形成可复制方法论。

**MVP In（本期做）**
- 博主库管理（CRUD + 标签 + 历史）
- Brief 录入 + AI 结构化生成
- 博主匹配推荐（Top-N + 理由）
- 内容工作台（AI 生成 + 合规校验）
- Campaign 管理 + 效果看板
- AI 复盘报告

**MVP Out（后续做）**
- 支付/结算自动化、复杂 RBAC、全套多语言、自动投放 API（依赖代理商接口）、多城市。

---

## 3. 用户角色

| 角色 | 权限 |
|---|---|
| Merchant（商户） | 提交需求、看推荐、看效果 |
| Creator（博主） | 维护主页、接单、交稿 |
| Operator（运营/我们） | 管理博主库、审核、匹配、投放协调 |
| Agent（代理商） | 投放/结算只读后台 |

---

## 4. 功能需求

### FR1 · 博主库管理
- 录入/编辑博主：`handle, platformId, followers, engagementRate, niche[], city, rateCAD, availability, pastCases[]`
- 列表 / 筛选 / 搜索；标签体系（赛道 / 调性 / 粉丝层级）
- 画像 embedding 向量预计算（用于匹配）

### FR2 · Brief 录入与 AI 生成
- 商户表单：行业、目标、预算、目标人群、卖点、KPI、时间
- AI（Claude，JSON 模式）生成结构化 Campaign Brief：目标人群包建议、内容方向、推荐博主类型、KPI 拆分
- 输出可编辑，存入 DB

### FR3 · 博主匹配
- 输入 Brief → 向量化 Brief 文本 + 博主画像 → cosine 相似度 + 业务权重（互动率、报价匹配度、城市）加权 → Top-N 推荐
- 每条推荐附**匹配理由**（标签命中 / 历史表现 / 报价匹配）

### FR4 · 内容工作台
- 选中博主 + Brief → AI 生成种草文案 / 脚本 / 标题 / 选题（多版本）
- 中英双语本地化；合规词表校验（违禁词 / 广告法极限词 / ASC 披露提示）
- 内容状态机：`草稿 → 待审 → 已发 → 复盘`

### FR5 · Campaign 与效果看板
- 创建 Campaign，绑定 Brief / 博主 / 内容 / 投放（代理商侧）
- 看板指标：曝光、互动、点击、留资、ROI（手动/半自动录入，后续接 API）
- 异常预警（如互动率骤降）

### FR6 · 复盘报告
- Campaign 结束 → AI 生成复盘（亮点、问题、下一轮建议）
- 报告沉淀进知识库（供 RAG 调用）

---

## 5. 非功能需求

- **性能**：博主库 < 1k 条时，列表/匹配响应 < 1s
- **安全**：商户/博主 PII 加密存储；密钥仅在服务端（Vercel 函数）
- **合规**：蒲公英为唯一付费合作通道；合作披露强制；PIPEDA 数据合规
- **可观测**：错误日志、AI 调用用量统计、限流

---

## 6. 验收标准

- [ ] 可录入 ≥ 30 个博主并成功匹配出一个 pilot Campaign
- [ ] Brief AI 生成可用，匹配 Top-5 合理且带理由
- [ ] 看板可展示核心指标（曝光/互动/留资/ROI）
- [ ] 复盘报告可由 AI 生成
